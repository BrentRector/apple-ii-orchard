# Decompiled SoftCard CP/M — complete distributions

This directory holds **complete, reassemble-to-byte-identical decompilations** of
the two Microsoft SoftCard CP/M releases studied in this project, each rendered as
commented assembly with an AI prose layer, plus tooling to rebuild the original
disk images from source.

| Distribution | Disk image | CP/M | Apple format |
|--------------|-----------|------|--------------|
| [`CPMV223-44K/`](CPMV223-44K/) | `CPMV223-44K.DSK` | **2.23** | DOS 3.3 order (`.dsk`) |
| [`CPMV220/`](CPMV220/) | `CPMV220-Disk1.po` | **2.20** | ProDOS order (`.po`) |

Each distribution is self-contained and identically organized:

```
<DISTRO>/
  os/           the operating system, as commented assembly
  utilities/    every .COM program in the filesystem, as commented Z-80 assembly
  README.md     the file inventory and per-distro notes
```

Shared tooling lives here at the top level:

```
generate_distribution.py <DISTRO>   (re)create the raw disassembly from the disk
verify_roundtrip.py [DISTRO ...]    reassemble every source file, compare to original bytes
rebuild.sh <DISTRO>                 rebuild the byte-identical disk image + verify
```

## The operating system

Both releases are the same architecture — a Z-80 CP/M hosted on a 6502 Apple II
through the SoftCard — so the `os/` layout is parallel across the two:

| Region | CPU | Role |
|--------|-----|------|
| `CPM_BootLoader` | 6502 | Stage-2 boot loader, install-copy logic, LOAD_CPM staging read |
| `CPM_RWTS` | 6502 | Read/Write Track-Sector engine (GCR 6-and-2 codec) |
| `CPM_InstallFragments` | 6502 | Fragments copied into place during install |
| `CPM_DiskCallbacks` | Z-80 | Thunks bridging BDOS/BIOS disk requests to the 6502 RWTS *(2.23 only)* |
| `CPM_SystemImage` | Z-80 | CCP (command processor) + BDOS |
| `CPM_BIOS` | Z-80 | BIOS: jump table + console / disk / IOBYTE primitives |

**The 2.20 → 2.23 difference** that this project traced is visible by diffing the
two `os/CPM_BIOS.asm`: 2.23 adds a Videx Videoterm device path (a device-6 / Pascal
1.1 console dispatch) that 2.20 lacks. See
[`../docs/CPM_Videx_Difference.md`](../docs/CPM_Videx_Difference.md) and
[`../docs/CPM_SoftCard_RealMap_Findings.md`](../docs/CPM_SoftCard_RealMap_Findings.md)
for the full analysis. `python -m cpm_pipeline diff <a> <b>` reports the delta
mechanically.

## The filesystem programs

`utilities/` is every executable (`.COM`) in each disk's CP/M directory, decompiled
to Z-80 source — the Digital Research tools (`ASM`, `DDT`, `ED`, `PIP`, `STAT`,
`LOAD`, `SUBMIT`, `XSUB`) and the Microsoft/SoftCard additions (`CPM60`/`CPM56`,
`MBASIC`, `GBASIC`, `FORMAT`, `COPY`, `BOOT`, `CAT`, …). Non-`.COM` directory
entries (tokenized `.BAS`, `.ASM` text) are not machine code and are not decompiled.

Decompilation is **emulation-assisted**: each program runs under the Z-80 core with
a CP/M BDOS shim to discover real code paths. Interactive programs (`MBASIC`,
`GBASIC`, `ED`, `PIP`, `STAT`) block on console I/O the shim doesn't model, so large
stretches come out as data — the code/data split is best-effort there, but every
file still round-trips byte-identical.

## Rebuild byte-identical

```bash
source ../../shared/toolchain/env.sh        # ca65 + ld65 + sjasmplus
bash rebuild.sh CPMV223-44K                      # -> CPMV223-44K/rebuilt/CPMV223-44K.DSK   (BYTE-IDENTICAL)
bash rebuild.sh CPMV220                       # -> CPMV220/rebuilt/CPMV220.po      (BYTE-IDENTICAL)
python verify_roundtrip.py                   # reassemble every source file, both distros
```

`rebuild.sh` reassembles the OS, reconstructs the disk image and byte-compares it
against the original, then reassembles every `os/` and `utilities/` file and confirms
each matches the original bytes — proving the annotated source is faithful (the
`[AI]` comments are assembler comments and never change a byte).

## How it was produced

The [`cpm_pipeline`](../cpm_pipeline/) toolchain — see [`../README.md`](../README.md):

```bash
python -m cpm_pipeline decompile-disk ../disks/CPMV223-44K.DSK out --ai
```

The `[AI]`-marked prose comments are generated by Claude `claude-opus-4-8`. With no
`ANTHROPIC_API_KEY`, the `--ai` flag uses the **Claude Code** CLI (`claude -p`),
which authenticates through Claude Code — no API key required. They are
machine-generated hints, not verified annotation.
