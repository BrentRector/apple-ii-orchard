# BASIC.asm provenance and the master-source switch

`BASIC.asm` is the **master source** for both `GBASIC.COM` and `MBASIC.COM`. As of the
promotion commit it is **edited directly** (by hand or by scripts that operate on
`BASIC.asm` itself); it is no longer regenerated by the decode/fold pipeline.

## The one invariant (the gate)

Every change to `BASIC.asm` must pass `test_fold_build_byte_identical`
(`cpm_pipeline/tests/test_utilities_roundtrip.py`): assemble `BASIC.asm` twice and diff
the output against the genuine `.COM`s on `softcard-cpm2.20-44k-system.dsk`:

```
source shared/toolchain/env.sh
cd softcard
python -m cpm_pipeline.basic.fold_build      # reports both modes BYTE-IDENTICAL or DIFFERS
```

- with the `GBASIC` symbol defined  -> `GBASIC.COM` (self-relocating, graphics-ON)
- without it                        -> `MBASIC.COM` (in place, graphics-OFF)

`test_basic_master.py` additionally lints the master + its includes for non-ASCII
(mojibake) bytes -- the byte gate ignores comments, so this is the guardrail that
direct editing otherwise loses (`apply_naming` used to sanitize comments automatically).

## Line addresses: the per-build listings

`BASIC.asm` no longer carries per-line `; $XXXX <bytes>` comments. They were redundant
with -- and less accurate than -- the assembler listing: a single inline address can
only be right for one build, but the GBASIC body relocates to `$3000` while MBASIC runs
flat at `$0100`. `fold_build` emits a per-build listing (address + machine code + source
per line) on every run:

```
python -m cpm_pipeline.basic.fold_build   # writes GBASIC.lst + MBASIC.lst next to BASIC.asm
```

`GBASIC.lst` / `MBASIC.lst` are the address reference; they are build artifacts
(git-ignored, regenerated). Semantic comments and inline notes were preserved in the
source; only the machine-generated address/byte tokens were removed.

## How `BASIC.asm` was originally derived (provenance-only)

The pipeline below DECODED the `.COM` bytes into relocatable source and FOLDED the two
builds into one conditional file. Both jobs are done; the chain is kept for provenance
and is **no longer the edit path**:

```
GBASIC.COM bytes
  -> cpm_pipeline.basic.gen_gbasic                 (clean-room decode -> GBASIC.asm, machine labels)
  -> cpm_pipeline.basic.apply_naming + GBASIC.overlay.json
                                                   (semantic names/comments/operand-rewrites; verifies == GBASIC.COM)
  -> cpm_pipeline.basic.fold_gen                   (GBASIC.asm + 11 IFDEF GBASIC patches -> BASIC.asm)
```

`fold_gen` is now **provenance-only**: it refuses to overwrite the committed `BASIC.asm`
unless forced with `--write-master`, because re-deriving from `GBASIC.asm` would discard
the direct edits made to the master since the switch:

```
python -m cpm_pipeline.basic.fold_gen                 # refuses (provenance-only)
python -m cpm_pipeline.basic.fold_gen --write-master  # full re-derive, DISCARDS direct edits
```

To reproduce/verify the clean-room derivation as it stood at the switch, check out the
promotion commit and run the chain (`gen_gbasic -> apply_naming --write -> fold_gen
--write-master -> fold_build`); it reproduces that commit's `BASIC.asm` and both `.COM`s.

## GBASIC.asm / MBASIC.asm

`GBASIC.asm` is the decode/fold **base** the provenance chain reads; `MBASIC.asm` (if
present) is the standalone graphics-OFF twin. After the switch they are **reference
views**, not the source of truth: their bytes still round-trip to their `.COM`s
(`test_utility_source_is_byte_identical`), but their annotations may lag `BASIC.asm`,
which is the file to edit. Regenerate them from the provenance chain if an up-to-date
single-build view is needed.
